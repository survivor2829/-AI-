## 坑 1：Playwright chromium 首次安装卡死 40 分钟零字节

### 症状

`docker compose build web` 跑到 `playwright install chromium` 这一步。日志冻在
"Downloading Chrome for Testing 145.0.7632.6 from https://playwright.azureedge.net/..."
40 分钟后仍零字节，BuildKit work dir 仅增 11M（都是解压前 5% 的 zip 头），外部 TCP 连接数为 0。

跑 `curl -I https://playwright.azureedge.net/` 从同一台服务器：HTTP 307，0.66 秒。
**网络本身通，但 playwright 的 node downloader 走 Azure CDN 大文件首次 TLS 握手 + backoff 死循环。**

### 根因

1. `pip install playwright` 走的是 pypi（已配腾讯云镜像），这部分没事
2. `playwright install chromium` 后续拉的 **150MB Chrome 二进制** 走的是 `playwright.azureedge.net`
3. 中国大陆访问 Azure Edge CDN 的大文件握手有时会卡在非常长的 backoff，不会超时、不会报错、不出任何进度

### 修法（迭代 4 次才稳）

**Commit 004cd51** — 第一次尝试：加 `PLAYWRIGHT_DOWNLOAD_HOST=https://npmmirror.com/mirrors/playwright`
→ **立刻 404**。npmmirror 的 OSS bucket key 是 `binaries/playwright/...` 而不是 `mirrors/playwright/...`，URL 写错了。
这次失败 10 秒，总比卡 40 分钟强。

**Commit 4ebbe94** — 第二次：改用 `https://cdn.npmmirror.com/binaries/chrome-for-testing`
→ Chrome + chrome-headless-shell（167M + 116M）下载成功（200 秒，8MB/s 稳）！
但 playwright 1.58 的 `install chromium` 会自动拉 ffmpeg 作依赖，ffmpeg 的 URL 结构完全不同：
`{HOST}/builds/ffmpeg/1011/ffmpeg-linux.zip` → `https://cdn.npmmirror.com/binaries/chrome-for-testing/builds/ffmpeg/...` → 404。
单个 DOWNLOAD_HOST 覆盖不了两个前缀。

**Commit 7fd3d4e** — 第三次：拆两阶段 RUN，`chromium 先 ffmpeg 后`，第一阶段 `|| true` 吃掉 ffmpeg 404
→ 依然失败。playwright 的 `INSTALLATION_COMPLETE` marker 是"整个 install 命令走完才落盘"，
第一阶段 ffmpeg 失败 → 整个命令非零退出 → chrome+shell 的 marker 没写 → 第二阶段换 HOST 重跑时又试图下 chrome-headless-shell → 404。

**Commit 19faf7b** — 第四次：反转顺序 `先 ffmpeg 后 chromium`
→ ✅ 成功。ffmpeg 2MB 秒装 + 独立 marker 落盘 → chromium 阶段看到 ffmpeg marker 已完成直接跳过 → chrome+shell 从 CfT 镜像下载 → 全绿。

### 立碑 —— 中国部署三铁律

**铁律 1**：凡是"pip/npm 装完再自动调第二个 downloader"的工具（playwright / puppeteer / electron-builder / node-gyp），都**必须额外配国内镜像**。pypi 镜像只管 Python 包，二进制资源是独立的下载链路。

**铁律 2**：单个 `PLAYWRIGHT_DOWNLOAD_HOST` 覆盖不了 `chrome-for-testing` + `ffmpeg` 两条路径。如果必须从同一镜像源拉两类资源，**分两次 RUN，每次改 env**。顺序关键：先装"小+单独"的（ffmpeg 2MB），后装"大+批量"的（chromium 167MB）。反过来大文件先跑、小文件炸掉会把大文件的 marker 带走。

**铁律 3**：失败快是好事，失败慢是噩梦。一旦看到下载卡住**超过 90 秒没进度 + 外部 TCP=0**，立刻 kill、换镜像，别等。40 分钟卡死的教训：一次浪费的时间够我迭代 8 次镜像配置。

### 锁死手段

Dockerfile 注释把"为什么要这么分两阶段"写成完整段落（见 commit `19faf7b` 的 11 行注释）。
下次改动者必须读懂注释才不会"优化"回单 RUN。

## 坑 2：Alembic 遇上 `db.create_all()` 老 SQLite 的双重初始化

### 症状

新镜像第一次启动后跑 `docker compose exec web flask db upgrade`：
```
sqlite3.OperationalError: table users already exists
  File "/app/migrations/versions/a73747e2b475_baseline_schema_from_models.py", line 21
    op.create_table('users', ...)
```

查一下 `alembic_version` 表：存在，但**内容为空**。
查一下实际表：`['alembic_version', 'batch_items', 'batches', 'generation_logs', 'users']` 全都有。

### 根因

老部署跑的 app.py 启动时有 `db.create_all()` 兜底，会把所有 model 定义的表创出来（不走 Alembic）。
升到新代码后，Alembic 的 baseline migration 又要从零 `create_table('users', ...)` —— 表已经存在，炸。
`alembic_version` 表是 Alembic 启动时自动建的容器，内容（`version_num` 列）才是 "哪个 migration 已跑" 的记录。空 = "我什么都没跑过"。

### 修法

```bash
flask db stamp a73747e2b475
```

`stamp` 告诉 Alembic "这个版本号当作已应用、不要真跑 upgrade SQL"。表结构已经 ok（从 `create_all` 来的和 baseline migration 生成的列完全一致，因为两者都从同一个 models.py 出），**只差一行 version 记录**。

### 立碑 —— Alembic 上线铁律

**铁律 4**：从 `db.create_all()` 过渡到 Alembic 的第一次部署，**必须 stamp baseline 而不是 upgrade**。判断条件：
- 目标表已存在（`inspect(db.engine).get_table_names()` 有它们）
- `alembic_version` 表 version_num 为空
→ 跑 `flask db stamp <baseline_revision>`，一次性补记录，不动表。

**铁律 5**：未来 models 加字段走 Alembic migration 时，**先确认 `db.create_all()` 已从启动流程里删掉**，否则新增列可能被两边都想创建，导致冲突或"哪边赢"不确定。
当前 app.py 还保留 `db.create_all()` 做 SQLite 开发兜底 —— 生产用 Postgres + migration 后，这个兜底要删或加"仅 dev 环境"判断。

## 坑 3：SSH 免密卡在 authorized_keys 格式错误

### 症状

第一次从本地 Windows SSH 到腾讯云：
```
Permission denied (publickey)
```

但公钥已经贴进 `/root/.ssh/authorized_keys`，`sshd -T | grep publickey` 显示 `pubkeyauthentication yes`，`chmod 600` 到位，本地 `~/.ssh/config` 也配了。

### 根因

通过腾讯云 Web 终端粘贴公钥时，heredoc 解析问题导致 `authorized_keys` 里公钥前面多了一段提示文字，使得一行变成：
```
# ... (复制下面的公钥...)ssh-ed25519 AAAAC3... user@host
```
sshd 解析时把整行当一个 key，前缀不是合法 key 类型 → 认不出 → 拒绝。

### 修法

用单引号 echo 一行命令重写，绕开 heredoc 变量展开：
```bash
echo 'ssh-ed25519 AAAAC3... user@host' > ~/.ssh/authorized_keys
```

### 立碑 —— SSH 配置排查三板斧

**铁律 6**：配完 SSH 免密第一次登录失败，按顺序查：
1. **`cat authorized_keys` 看 raw bytes**：公钥必须一行，开头必须严格匹配 `ssh-` 或 `ecdsa-` 或 `sk-ssh-`。任何前缀注释会让整行失效。Web 终端粘贴时最容易带脏字符。
2. **`sshd -T | grep -E 'publickey|authorized'`**：确认 sshd 真的启用了 publickey 且读的是预期路径。别信 `/etc/ssh/sshd_config` 里的注释状态 —— `sshd -T` 是真的生效值。
3. **`ls -la ~/.ssh/ && ls -la ~/`**：权限必须 `700 ~/.ssh`，`600 authorized_keys`，`~/` owner 必须是目标用户。任何一处给 world-writable 都会被 sshd 拒绝。

**立即写 `~/.ssh/config` 起别名**：把 `Host <alias>` + `HostName <ip>` + `IdentityFile` 配好，之后所有命令走别名（`ssh tencent-prod` 比 `ssh root@124.221.23.173 -i ~/.ssh/id_ed25519` 省时省错）。Windows 下用 PowerShell `Out-File -Encoding ascii` 避免 UTF-8 BOM。

## 坑 4 (遗留，未处理，仅归档)：sshd PasswordAuthentication yes

### 现状

腾讯云默认 `/etc/ssh/sshd_config` 里 `PasswordAuthentication yes`。密钥登录已能工作，但密码登录通道仍开着 —— 意味着攻击者可以无限尝试 root 密码爆破。

### 未立即收紧的原因

本次部署主线是"让服务跑起来"，不想在半夜动 sshd 把自己锁在门外。密钥登录验证通过、`~/.ssh/config` 别名可用、重启 docker 不依赖 ssh 密码 —— 收紧是低优先级、可延后的硬化动作。

### 待办（阶段七头几天做）

1. 本地再开一个 ssh 会话保持连接（防锁门）
2. `sed -i 's/^#\?PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config`
3. `systemctl restart ssh`
4. 在保持连接的会话里 `exit` + 新开会话验证仍能免密登录
5. 若失败：在保持连接的会话里改回 `yes` + 重启 ssh

## 横向教训（跨 4 个坑的共性）

### 1. "看起来能跑" ≠ "真的能跑"

- `curl -I https://playwright.azureedge.net/` 返回 307，**网络通**。但大文件下载会卡死 —— 握手通不代表带宽通。
- `alembic_version` 表存在，**Alembic 初始化了**。但内容为空 —— 存在不代表记录。
- 公钥贴进了 authorized_keys，**文件存在**。但多了前缀 —— 存在不代表格式对。

**判据换一遍**：别看"是不是有"，看"是不是按预期工作"。对每一个环节问"我怎么证明它真的在按预期工作"，答案必须是命令 + 预期输出的组合。

### 2. 中国大陆部署的"默认国外资源"陷阱

Docker / Python / Node 工具链默认从海外拉东西，在国内都得过一层镜像：
- `apt` → 腾讯云/阿里云镜像（已配）
- `pip` → 腾讯云/清华镜像（已配）
- `playwright install` → npmmirror CfT（这次踩坑补上）
- `docker pull`（基础镜像）→ 可选配置 `registry-mirrors`（腾讯云节点已经够快，暂不换）
- rembg model（`.onnx` 文件）→ 已用 `ghfast.top` 镜像前置加速

**凡是有独立 downloader 的工具**：`playwright`、`puppeteer`、`electron`、`chromedriver-manager`、`selenium-manager`、`browsers.json` 风格的任何东西 → **都得独立配镜像**。别指望一个顶层 env var 解决全部。

### 3. 生产上线 + 存量数据库的 "第一次 migration" 必定出坑

ORM 从 `create_all` 过渡到 Alembic 几乎一定会遇到 baseline 冲突。下次类似升级的预检：
1. `inspect(db.engine).get_table_names()` → 列出所有表
2. `SELECT version_num FROM alembic_version` → 看是否为空
3. 对比 `migrations/versions/` 里的 baseline → 两边表集合是否一致
4. 一致 → `stamp baseline`；不一致 → 手工写 conditional migration 或 manual ALTER
